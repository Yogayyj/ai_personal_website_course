第3课：GitHub 工作流程实战

让学生从“会改网页”升级为“会用标准开发流程管理网页”。

本节课学生会完成

  • 理解 GitHub 工作流程的完整路径
  • 弄清楚 commit、branch、pull request 的含义
  • 学会在分支上修改并提交代码
  • 学会发起 PR 并合并到主分支
  • 触发 GitHub Pages 自动更新线上网站

核心概念讲解（课堂重点）

  • Repository（仓库）：项目总文件夹，保存网站代码和历史记录。
  • Branch（分支）：并行开发线。main 是稳定版本，其他分支用于开发新内容，不会直接影响正式站点。
  • Commit（提交）：一次“有说明的保存快照”。每次 commit 都应只做一类修改，并写清楚改了什么。
  • Pull Request（PR）：把自己分支的改动申请合并进主分支，并让老师/同学 review。
  • Merge（合并）：PR 通过后，把分支改动并入 main。
  • GitHub Pages：读取仓库中的网页文件，自动生成线上可访问的网站。

标准流程（建议全班统一）

  1. 从 main 创建新分支（例如 feature/update-homepage）。
  2. 在新分支中修改 index.html 或其他页面文件。
  3. 本地预览，确认页面显示正常。
  4. 提交 commit（示例：feat: update homepage hero section）。
  5. push 到 GitHub，发起 Pull Request。
  6. 完成 review 后 merge 到 main。
  7. GitHub Pages 自动构建并发布新版本网站。

Commit 怎么写才专业？

  • 建议结构：类型: 修改内容（例如 feat: add projects section）。
  • 常见类型：feat（新功能）、fix（修复）、docs（文档）、style（样式）。
  • 一条 commit 只做一件事，便于回滚和排错。

课堂练习

  • 每位同学创建一个自己的分支并命名。
  • 修改首页一句自我介绍并提交 commit。
  • 发起一个 PR，请同桌进行 review。
  • 完成 merge 后，检查 Pages 链接是否更新。

课后任务

  • 为个人网站再完成 1 次 branch → commit → PR → merge 流程。
  • 整理 3 条最清晰的 commit message 截图，作为学习记录。
  • 记录一次自己遇到的冲突或报错，并写出解决方法。
